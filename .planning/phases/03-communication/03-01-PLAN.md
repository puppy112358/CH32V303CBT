# Plan 01: Protocol Module Scaffold + USART2 Init + Ring Buffer + ISR

## Objective

Create the `Drivers/protocol.c/.h` module scaffold with USART2 (PA2=TX, PA3=RX) initialization at 115200 bps 8N1+odd parity, a 512-byte interrupt-driven RX ring buffer, the `USART2_IRQHandler` ISR that pushes bytes and handles UART errors, and a `protocol_poll()` function to extract newline-delimited lines at the start of each control cycle. Wire `protocol_init()` into main.c's initialization sequence. This establishes the physical communication channel — COMM-01 RX path foundation.

## must_haves

- `Drivers/protocol.h` with include guard `__PROTOCOL_H`, C++ extern block, public API: `protocol_init()`, `protocol_poll()`, `protocol_send()`
- `Drivers/protocol.c` with USART2 init at 115200 bps, 8 data bits, 1 stop bit, odd parity, RX+TX mode
- 512-byte static volatile ring buffer (`rx_buf[512]`) with `rx_head`/`rx_tail` indices
- `USART2_IRQHandler` in ch32v30x_it.c: RXNE pushes byte, handles parity/framing/overrun errors
- `protocol_poll()`: scans ring buffer for `\n`, copies complete line to static buffer, returns pointer (or NULL)
- `protocol_init()` called from main.c after USB-CDC init, before the super-loop
- USART2 RXNE interrupt enabled at NVIC priority 0x02 (lower than EXTI4 at 0x01)
- Follows existing Drivers/ module conventions: lowercase filenames, `#include "../Drivers/protocol.h"` pattern

## truths

- "UART0" in project documentation = USART2 hardware peripheral (APB1 at 0x4400 offset, IRQ #54 `USART2_IRQn`)
- USART1 is already used for debug printf on PA9 — must not be repurposed
- PA2=TX (AF push-pull), PA3=RX (input floating) — no conflict with I2C1 (PB6/PB7), EXTI4 (PA4), or debug USART1 (PA9)
- Baud rate divider at 96 MHz PCLK1: IntegerDivider ≈ 52.08, fractional = 1 → actual baud ≈ 115,246 bps (0.04% error)
- With odd parity + 8 data bits, each byte = 11 bits on wire (1 start + 8 data + 1 parity + 1 stop)
- Blocking TX pattern follows `_write()` in debug.c: `while(USART_GetFlagStatus(USART2, USART_FLAG_TC) == RESET); USART_SendData(USART2, byte);`
- 512-byte ring buffer holds ~4–5 typical commands (50–100 bytes each) at 115200 bps
- ISR does minimal work (read RDR, write ring buffer, increment tail) — no printf in ISR
- UART errors (PE/FE/ORE) are counted in volatile flags and reported by main loop via printf

## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| USART2 RX buffer overflow from sustained high-rate input | Low | 512B ring buffer holds ~4 commands; rx_overflow flag triggers flush + error ack; PC must wait for ack before next command |
| Parity error causes bit corruption in command | Low | Parity error detected in ISR, affected byte discarded; incomplete command fails JSON parse → error ack |
| USART2_IRQHandler blocking (I2C or printf in ISR) | High | ISR design explicitly limits to RDR read + ring buffer write + flag checks; no blocking calls. Code review verification. |
| USART2 IRQ priority inversion with EXTI4 | Medium | USART2 at priority 0x02, EXTI4 at 0x01 — fault ISR always preempts UART ISR. Verified by priority configuration. |
| PA2/PA3 pin conflict with other peripherals | Low | Verified: PA2/PA3 are USART2 default pins; not used by I2C1 (PB6/PB7), EXTI4 (PA4), or debug USART1 (PA9) |
