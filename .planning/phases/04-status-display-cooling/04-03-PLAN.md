# Plan 03: Fan PWM + Tachometer + PID + Stall Detection — TIM3 + PID Reuse

## Objective

Create the `Drivers/fan.c/.h` module that controls the cooling fan via TIM3 CH1 PWM (PA6, 25kHz), measures RPM via TIM3 CH2 input capture (PA7, tachometer period measurement), runs a PID controller using the existing `Drivers/pid.c/.h` API with a new fan instance targeting 50°C heatsink temperature, and detects fan stall (RPM=0 with PWM duty ≥ 25% for 2 seconds). Add the `TIM3_IRQHandler` for input capture to `ch32v30x_it.c`. Add a fan stall soft-warning flag to the fault register. Wire init and loop updates into main.c. This delivers FAN-01 (PWM speed control), FAN-02 (tachometer RPM measurement), and FAN-03 (PID with anti-windup).

## must_haves

- `Drivers/fan.h` with include guard `__DRIVERS_FAN_H`, C++ extern block, public API: `fan_init()`, `fan_update()`, `fan_get_rpm()`, `fan_get_duty_pct()`, `fan_is_stalled()`
- `Drivers/fan.c` with TIM3 CH1 init (PA6, PWM output, 25kHz), TIM3 CH2 init (PA7, input capture, rising edge, period measurement), fan PID instance, stall detection state machine
- Fan PID: new `PID_Instance fan_pid` with coefficients `FAN_KP=0.2f`, `FAN_KI=0.02f`, `FAN_KD=0.0f` (PI-only initially, avoids derivative noise on temperature measurement). PID setpoint = 50.0°C (FAN_TARGET_TEMP_C). PID feedback = `heatsink_temp_c`. PID dt = 0.1f (100ms).
- Fan duty: OFF (0%) when temp < 50°C. Active range: 30-100% PWM duty when temp ≥ 50°C. PID output maps linearly to duty in active range.
- TIM3 CH1 PWM: 25kHz (PSC=71, ARR=39 at 72MHz → 72MHz/72/40 = 25kHz). Duty range: CCR1 = 0 (0%, OFF), CCR1 = 12-39 (30-100%, active).
- TIM3 CH2 input capture: rising edge, no prescaler filter. Period between consecutive captures → RPM = 30,000,000 / period_ticks (TIM3 at 1MHz with PSC=71).
- Tachometer global: `volatile uint16_t fan_rpm = 0;` updated by ISR/main loop
- Stall detection: 20 consecutive 100ms cycles (2 seconds) with RPM=0 and duty ≥ 25%. Sets `fan_stall` flag in a new `fan_status` bitfield or soft flag. Clears immediately when RPM > 0 detected.
- Fan stall flag adds a non-latching soft warning — telemetry and printf only, no system state change. Hard thermal shutdown is v2.
- `fan_init()` called from main.c init after temp_sensor_init(), before super-loop
- `fan_update()` called from main.c super-loop each 100ms cycle: compute PID, update PWM, check stall
- `TIM3_IRQHandler` in ch32v30x_it.c: handles CH2 input capture (CC2 interrupt) for tachometer period measurement
- `fan_stall_flag` extern visible to protocol.c for future telemetry (Plan 03 task adds the field)
- Follows existing Drivers/ module conventions

## truths

- PA6 = TIM3 CH1 (PWM output) — confirmed free on LQFP48. PA7 = TIM3 CH2 (input capture) — confirmed free.
- No pin conflicts: PA0=TIM2 CH1 (WS2812), PA2/PA3=USART2, PA4=EXTI4, PA5=ADC1 CH5 (NTC), PA6/PA7=TIM3 CH1/CH2 (fan), PA9=USART1, PA11/PA12=USB, PB6/PB7=I2C1
- TIM3 clock = APB1 (PCLK1) at 72MHz (APB1 prescaler /2 from 144MHz, but TIM3 clock doubles to 144MHz when APB1 prescaler ≠ 1 per CH32V303 clock tree)
- **CRITICAL:** CH32V303 TIM3 is on APB1. When APB1 prescaler is /2 (as it is with 96MHz HSE → 72MHz PCLK1), the TIMx clock = PCLK1 × 2 = 144MHz. This means:
  - TIM3_Prescaler = 143 → TIM3 counter = 144MHz / 144 = 1MHz
  - For 25kHz PWM: ARR = 1,000,000 / 25,000 - 1 = 39
  - For tachometer: at 1MHz counter, 1 tick = 1μs. RPM = 60,000,000 / (period_ticks × 2) for 2 pulses/rev.
    - Actually: RPM = (60 sec/min) / (period_seconds × 2 pulses/rev) = 60 / (period_us/1e6 × 2) = 30,000,000 / period_us
    - At 1MHz: period_ticks = period_us, so RPM = 30,000,000 / period_ticks
    - Example: 1500 RPM → period_us = 30,000,000/1500 = 20,000μs per 2 revs? No...
    - Let me re-derive: 1 revolution = 2 tacho pulses. At 1500 RPM: 1500/60 = 25 rev/sec. 25 × 2 = 50 pulses/sec. Period = 1/50 = 20,000μs.
    - At 1MHz counter: period_ticks = 20,000. RPM = 30,000,000 / 20,000 = 1500. ✓ Correct.
- Fan PWM frequency: 25kHz (standard Intel 4-wire PWM fan spec). Duty cycle resolution at ARR=39: ~2.5% per step (1/40). Fine enough for fan control (30% min to 100% max = 28 steps).
- Input capture: T2 rising edge, CC2 interrupt. ISR reads TIM_GetCapture2(TIM3), computes delta from previous capture, stores in `tacho_period_ticks`, sets valid flag.
- Stall detection state: `static uint8_t stall_counter = 0;` — incremented each 100ms cycle when fan_rpm == 0 AND fan_duty_pct >= 25. Reset to 0 when fan_rpm > 0. Stall confirmed when counter >= 20 (2 seconds).
- Fan PID output range: PID output 0 to FAN_PID_MAX=100.0 maps to PWM CCR1 range: OFF (0) or 30-100% of ARR (12-39 of 40 total at ARR=39).
- Fan PID reuse: `pid_init(&fan_pid, FAN_KP, FAN_KI, FAN_KD);` — same API as CV/CC PID. Anti-windup built-in. Integral freezes at output saturation (100% or 30% duty).
- No tachometer input pull-up needed: PC fan tachometer is open-drain with internal pull-up on fan side. Configure PA7 as GPIO_Mode_IN_FLOATING (TIM input capture uses the GPIO input path, not AF). Actually: TIM3 CH2 input capture requires PA7 alternate function. Use GPIO_Mode_IPU for weak pull-up (some fans need it).

       - stall_cycle_counter = 0;
       - If fan_stall == 1: printf("[FAN] Stall cleared — RPM=%d\r\n", fan_rpm);
       - fan_stall = 0;

## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Fan stalls (RPM=0) while temp > 50°C → no cooling → MOSFETs overheat | Medium | Stall detection sets fan_stall=1 within 2 seconds. Telemetry reports stall. MOSFET thermal mass provides time buffer (heatsink absorbs heat). v2 adds hard thermal shutdown at 80°C. |
| Fan PWM stuck at 0% (software bug or TIM3 not enabled) → fan never runs | Medium | `fan_update()` explicitly maps temp ≥ 50°C to 30-100% duty. If PWM stays at 0 despite PID output > 0, this is a code bug caught during verification (oscilloscope on PA6). |
| TIM3 tachometer period overflow at very low RPM | Low | At 1MHz timer and minimum plausible fan RPM of 300: period = 30e6/300 = 100,000 ticks > 65535 (16-bit overflow). Solution: use 32-bit `tacho_last_capture` and `tacho_period_ticks` — 32-bit subtraction handles 16-bit counter wraparound naturally. Max period at 300 RPM: 100,000 fits in 32-bit. |
| Fan PID integral windup during startup from cold | Low | Anti-windup in pid_compute() freezes integral when output saturated at 100% (FAN_PID_MAX). Once temp drops below 50°C, integral has not accumulated excess. Verified by existing anti-windup logic. |
| TIM3 CH2 input capture interrupt storm from noisy tachometer signal | Low | Tachometer signal is clean digital (hall-effect sensor in fan) at 25-200Hz. No debouncing needed. If noise: TIM_ICFilter can be increased from 0 to 0xF (digital filter) — configurable without code change. |
| Fan PID oscillation due to temperature measurement delay | Low | System thermal time constant is minutes. 100ms PID cycle is fast relative to thermal dynamics. PI-only (Kd=0) avoids derivative noise amplification. Oscillation unlikely with conservative Kp=0.2, Ki=0.02. |
| PA6/PA7 pin conflict with other planned uses | Low | Verified: PA6 and PA7 are TIM3 CH1 and CH2 default pins. No other peripherals assigned to these pins in LQFP48 package for this project. |
