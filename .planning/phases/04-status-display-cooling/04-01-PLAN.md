# Plan 01: WS2812 LED Status Display — TIM2 PWM + DMA Bitstream Driver

## Objective

Create the `Drivers/ws2812.c/.h` module that drives 2 WS2812 LEDs via TIM2 CH1 PWM (PA0) + DMA1 Channel 5 at 800kHz. Pre-compute a 48-byte bitstream buffer encoding GRB colors for both LEDs with 50% brightness. Map the system state (`SystemMode` enum → green/cyan/blue/red) and update LEDs only on state change. Wire initialization into main.c and add the TIM2 DMA TC interrupt handler to ch32v30x_it.c. This delivers LED-01 (DMA-driven zero-CPU transmission) and LED-02 (state-indicating colors).

## must_haves

- `Drivers/ws2812.h` with include guard `__DRIVERS_WS2812_H`, C++ extern block, public API: `ws2812_init()`, `ws2812_update_from_mode(SystemMode mode)`, `ws2812_set_color(uint8_t r, uint8_t g, uint8_t b)`
- `Drivers/ws2812.c` with TIM2 CH1 init at 800kHz (PSC=0, ARR=89, PWM mode 1), DMA1 Channel 5 init (Memory→Peripheral, byte-sized, 48 transfers), 48-byte static bitstream buffer
- Bitstream encoding: each byte maps one NRZ bit to PWM CCR value — T0H=25 (bit 0, ~28% duty), T1H=50 (bit 1, ~56% duty) at ARR=89
- WS2812 GRB color order correct: green byte first, then red, then blue
- 50% brightness: each RGB byte halved (`r/2`, `g/2`, `b/2`) before bitstream encoding
- State→color mapping: MODE_IDLE→green(0,32,0), MODE_CV→cyan(0,32,32), MODE_CC→blue(0,0,32), MODE_FAULT→red(32,0,0) — all at 50% brightness
- `ws2812_update_from_mode()` only recomputes bitstream when `system_mode` changed since last call — tracks `static SystemMode last_mode`
- DMA configured as single-shot (not circular): triggered once per update, 48 transfers, TC flag checked before next update
- `TIM2_IRQHandler` in ch32v30x_it.c: on DMA TC interrupt, disable TIM2 and DMA to stop transmission + ensure RESET condition
- `ws2812_init()` called from main.c init section after PID init, before super-loop
- `ws2812_update_from_mode(system_mode)` called from main.c super-loop after step 5 (state machine dispatch)
- Follows existing Drivers/ module conventions: lowercase filenames, `#include "../Drivers/ws2812.h"`, SPL init struct pattern

## truths

- TIM2 CH1 on PA0 (default mapping, no remap) — confirmed free: only PA4 (EXTI4), PA2/PA3 (USART2), PA9 (USART1), PA11/PA12 (USB), PB6/PB7 (I2C1) are in use
- DMA1 Channel 5 is the hardware request channel for TIM2_CH1 (CH32V303 DMA request mapping) — confirmed no other DMA usage in the project
- TIM2 clock source = APB1 (PCLK1) at 72MHz when APB1 prescaler ≠ 1 (96MHz HSE → 72MHz APB1 with /2 prescaler)
- With PSC=0 and ARR=89: TIM2 counter at 72MHz, period=90 cycles → 800kHz PWM frequency. T0H=25 cycles → 0.347μs, T1H=50 cycles → 0.694μs. Both within WS2812B timing tolerance (±150ns)
- DMA transfer direction: Memory→Peripheral, MemoryBaseAddr = bitstream buffer, PeripheralBaseAddr = (uint32_t)&TIM2->CH1CVR, 48 bytes, memory increment ON, peripheral increment OFF
- WS2812 RESET code (>50μs low) achieved by: after DMA completes, TIM2 output goes to idle state (CCR=0 or TIM disabled). The 100ms control cycle guarantees >50μs gap between updates.
- PWM output polarity: active high (TIM_OCPolarity_High) — CCR value = high pulse duration
- 48-byte buffer = 2 LEDs × 24 bits/LED (8 green + 8 red + 8 blue per LED, GRB order)
- GPIO PA0 alternate function: push-pull output for TIM2 CH1. `GPIO_Mode_AF_PP`, `GPIO_Speed_50MHz`
- No TIM2 interrupt enable needed for the CPU — only DMA TC interrupt to signal completion. TIM2 itself runs autonomously via DMA.

## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| DMA1 Channel 5 configuration corruption during transmission | Low | DMA is single-shot; each update re-configures CNTR and MADDR after DISABLE. DMA_Cmd DISABLE + reconfigure + ENABLE sequence is atomic from CPU perspective. |
| TIM2 output stuck HIGH if DMA fails to complete | Low | DMA TC timeout can be detected by ws2812_dma_busy flag not clearing; a future task could add a timeout watchdog in main loop (deferred). WS2812 spec allows indefinite HOLD without damage. |
| Simultaneous TIM2 and I2C1 DMA (DMA1 shared bus) | Low | DMA1 has internal priority arbiter (DMA_Priority_High). I2C1 uses no DMA currently; even if added, 48-byte WS2812 transfer completes in ~60μs — negligible contention. |
| Wrong color displayed due to GRB/RGB byte order swap | Medium | Verified by code review: colors[3] = {g, r, b} explicitly. Test via visual inspection on hardware. |
| Brightness miscalculation (too bright or too dim) | Low | 50% = integer division by 2. If LEDs appear wrong, macro WS2812_BRIGHTNESS_DIV is tunable at a single location. |
| PA0 pin conflict with other uses | Low | Verified: PA0 is TIM2 CH1 default pin, not used by any existing peripheral (I2C1=PB6/PB7, USART1=PA9, USART2=PA2/PA3, EXTI4=PA4, USB=PA11/PA12). |
